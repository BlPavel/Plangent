# План: Библиотека скиллов, команд и инструкций для агентов

> Этот файл — самодостаточное ТЗ. Его можно загрузить в новую сессию нейронки и выполнить.
> Проект: Plangent (AI-оркестратор). Stack: Node.js 22 / TypeScript / Express / better-sqlite3 / node-pty / ws; фронт Vue 3 + Pinia + Vite.

## Цель

Заменить текущий примитивный механизм скиллов (один склеенный `.md` файл) на полноценную **Библиотеку** управляемых сущностей, которые Plangent умеет раскладывать под разные нейронки (Claude Code, Codex и любые будущие CLI), на двух уровнях (глобальный + проектный), с поддержкой динамической загрузки.

## Контекст: как это работает (важно понять перед реализацией)

Агенты (Claude Code, Codex) при старте читают скиллы/команды/главный файл из **определённых папок на диске**. Plangent держит **Библиотеку как источник правды** и **синхронизирует** её содержимое в эти папки: при сохранении итема — пишет файлы во все папки указанных агентов; при удалении — убирает их оттуда. Файлы лежат на диске **постоянно** (не деплоятся/чистятся на каждый запуск). Агент при старте просто видит то, что уже синхронизировано.

Это сознательно простая модель (выбрана пользователем): нет манифестов на сессию, нет хуков на закрытие терминала, нет привязки к `launchAgent`. Один механизм — `sync()` — приводит папки в соответствие текущей Библиотеке.

### Ключевой факт: форматы Claude и Codex сошлись (актуально на июнь 2026)

Оба используют **один формат `SKILL.md`**: YAML frontmatter (`name` + `description`) + markdown-тело + прогрессивная (динамическая) загрузка — в контекст сначала идут только `name`+`description`+путь, полное тело грузится, когда агент решает применить скилл. Различаются **только пути на диске**:

| Сущность | Claude Code | Codex |
|---|---|---|
| Скилл (проект) | `{repo}/.claude/skills/<name>/SKILL.md` | `{repo}/.agents/skills/<name>/SKILL.md` |
| Скилл (глобально) | `~/.claude/skills/<name>/SKILL.md` | `~/.agents/skills/<name>/SKILL.md` |
| Команда (проект) | `{repo}/.claude/commands/<name>.md` | (как скилл; имя → `/slash`) |
| Главный файл (проект) | `{repo}/CLAUDE.md` | `{repo}/AGENTS.md` |
| Главный файл (глобально) | `~/.claude/CLAUDE.md` | `~/.codex/AGENTS.md` |

Frontmatter скилла:
```yaml
---
name: skill-name            # имя; для команд становится /slash-командой
description: Когда применять этот скилл (и когда НЕ применять).
disable-model-invocation: true   # опц. — только ручной вызов (= "команда")
---
Тело инструкции в markdown.
```

## Принятые решения (зафиксированы с пользователем)

1. **Механизм доставки — КОПИРОВАНИЕ + постоянные файлы (sync-модель).** Не symlink. Не deploy-on-launch. Файлы пишутся в папки агентов при сохранении итема и удаляются при удалении итема; лежат постоянно. Причины: соответствует ручному workflow пользователя (он и так держит `.claude` в репо); работает в Docker/Windows; нет хрупкой логики очистки на каждую сессию.
2. **Уровни (scope): глобальный + проектный.** Глобальные → `~/.claude`/`~/.agents` (для всех проектов). Проектные → `{repo}/.claude`/`.agents` конкретного проекта.
3. **Sync — единый механизм для обоих уровней.** Вызывается при создании/правке/удалении итема и по кнопке «Синхронизировать». НЕ привязан к запуску агента. Агент при старте просто видит уже синхронизированные файлы.
4. **Команды и скиллы — раздельные типы** в UI и хранилище (хотя под капотом близки).
5. **Главный файл (CLAUDE.md / AGENTS.md)** — отдельный тип сущности.
6. **Вложения/ресурсы** рядом со скиллом (scripts/, templates/) — поддержать.
7. **Per-agent override контента — опционально.** По умолчанию один `SKILL.md` для всех агентов; возможность переопределить тело под конкретного агента.
8. **Удаление — только свои файлы.** Plangent помечает свои файлы префиксом `plangent-` (или изолирует в свою под-папку) и при удалении/ресинке трогает ТОЛЬКО их. Существующую ручную `.claude`/`.agents` пользователя не затрагивает никогда.
9. **Видимость в git — настраивается на уровне проекта (галочка «Скрыть от git»).** Механизм — `{repo}/.git/info/exclude` (НЕ `.gitignore`): локальный файл внутри `.git/`, не пушится и не виден другим разработчикам (в отличие от `.gitignore`, который выдал бы факт использования нейронки). При включении Plangent дописывает туда свои пути идемпотентно (между маркерами `# >>> plangent` / `# <<< plangent`), при выключении — убирает их. Работает только для untracked-файлов — наши файлы изначально untracked, так что подходит. По умолчанию опция ВЫКЛючена (файлы видимы, можно шарить с командой).
10. **Объём: сразу бэкенд + фронтенд** (раздел «Библиотека» в UI).

## Целевая структура хранилища

```
data/library/
  skills/<slug>/
    SKILL.md                  # основной (общий для всех агентов)
    overrides/<agentType>.md  # опц. override тела под конкретного агента
    assets/...                # опц. вложения
  commands/<slug>/
    COMMAND.md
  main/
    <slug>.md                 # главные файлы (CLAUDE.md/AGENTS.md контент)
```

Метаданные (scope, привязка к проектам, frontmatter, к каким агентам применять) — в БД. Контент — на диске (удобно версионировать и редактировать).

## Карта путей на тип агента (layout profile)

В записи агента добавить поле `layout_profile` (JSON), описывающее куда класть файлы. Дефолты:

```jsonc
// claude
{
  "skills":   { "dir": ".claude/skills",   "global": "~/.claude/skills",   "file": "<name>/SKILL.md" },
  "commands": { "dir": ".claude/commands",  "global": "~/.claude/commands", "file": "<name>.md" },
  "main":     { "file": "CLAUDE.md",        "global": "~/.claude/CLAUDE.md" }
}
// codex
{
  "skills":   { "dir": ".agents/skills",    "global": "~/.agents/skills",   "file": "<name>/SKILL.md" },
  "commands": { "dir": ".agents/skills",    "global": "~/.agents/skills",   "file": "<name>/SKILL.md", "asSkill": true },
  "main":     { "file": "AGENTS.md",        "global": "~/.codex/AGENTS.md" }
}
```

Для нового CLI пользователь задаёт свой профиль через UI агента. Если у агента нет нативных скиллов — `asMerged: true` (склеить всё в главный файл, как старое поведение).

## Этапы реализации

### Этап 1 — Схема БД и storage
- Новая таблица `library_items`:
  - `id`, `type` (`skill`|`command`|`main`), `slug`, `title`, `description` (для frontmatter),
  - `scope` (`global`|`project`), `project_id` (NULL для global),
  - `frontmatter` (JSON: доп. поля вроде `disable-model-invocation`),
  - `agent_filter` (JSON: список agentId/типов, к которым применять; пусто = ко всем),
  - `enabled` (bool), `created_at`, `updated_at`.
- Новая таблица `library_overrides`: `item_id`, `agent_type`, путь к файлу override.
- Добавить агенту поле `layout_profile TEXT` (JSON) в `server/db/schema.ts` + миграция (см. `migrateData`).
- Добавить проекту флаг `hide_from_git` (bool, по умолчанию 0) — управляет записью в `.git/info/exclude`.
- Засеять дефолтные `layout_profile` для `agent-claude` и `agent-codex`.
- Storage-модуль `server/storage/library.ts`: CRUD по `library_items`.
- Обновить `server/models/types.ts`: интерфейсы `LibraryItem`, `LayoutProfile`, расширить `Agent`.

### Этап 2 — Library manager (замена skills/manager.ts)
- `server/skills/library-manager.ts`:
  - чтение/запись контента итемов на диск (`data/library/...`),
  - парсинг и валидация YAML frontmatter (взять зависимость `gray-matter` или мини-парсер),
  - управление вложениями (`assets/`),
  - управление override-файлами.
- Сохранить обратную совместимость или мигрировать старые `data/skills/common.md` + `data/skills/projects/*.md` в новые `library_items` (скилл-итемы). Написать одноразовую миграцию.

### Этап 3 — Sync-движок (копирование, постоянные файлы)
- `server/skills/syncer.ts`:
  - `resolveLayout(agent)` → берёт `layout_profile` или дефолт по имени/команде.
  - `targetsForItem(item, agent)`: вычислить целевые пути для итема под конкретного агента и scope (глобальный → `~/...`; проектный → `{repo}/...`). Выбрать тело (override под agent_type или основной).
  - `syncAll()` — основной механизм («привести всё в соответствие Библиотеке»):
    1. собрать все enabled-итемы (глобальные + проектные по всем проектам);
    2. для каждого активного агента, к которому применим итем (`agent_filter`), записать файл в целевой путь; создать недостающие папки;
    3. удалить **наши** файлы (с префиксом `plangent-` / в нашей под-папке), которых больше нет в Библиотеке (сняты/выключены/удалены);
    4. **никогда** не трогать чужие файлы пользователя.
  - `syncItem(item)` / `unsyncItem(item)` — точечный синк/удаление одного итема (для быстрых операций add/edit/delete без полного прохода).
  - Идемпотентность: повторный вызов не плодит дубли и не ломает существующее.
  - Защита от коллизий: имена/папки с префиксом `plangent-`; если в целевом пути уже лежит чужой файл (не наш) — не перезаписывать, предупредить.
  - **Нет** функций deploy-on-launch и cleanup-on-exit — модель постоянная.
  - `applyGitExclude(project)` / `removeGitExclude(project)`: если у проекта включён флаг «Скрыть от git» — дописать пути в `{repo}/.git/info/exclude` идемпотентно между маркерами `# >>> plangent` / `# <<< plangent`; при выключении — убрать блок. Вызывается из sync. Проверять наличие `.git/` (если репо не git — пропустить).
- Удалить мёртвый `deploySkillsToTemp` из старого менеджера.

### Этап 4 — Триггеры синхронизации
- Синк вызывается из API при мутациях Библиотеки (Этап 5): create/update/delete → `syncItem`/`unsyncItem`; ручная кнопка → `syncAll`.
- Дополнительно — `syncAll()` при старте сервера (в `server/index.ts`), чтобы папки всегда соответствовали БД (на случай ручных правок/переноса машины).
- **Запуск агента трогать НЕ нужно** — `runs.ts` и `terminal.ts` остаются как есть в части скиллов; убрать из `runs.ts` старый вызов `deploySkillsToProject` и связанную с ним очистку (`cleanupSkillsFromProject`, поле `skillsFile` в `activeSessions`).
- Удалить из `server/adapters/generic.ts` функции `deploySkillsToProject` и `cleanupSkillsFromProject` (заменены sync-движком).

### Этап 5 — REST API
- `server/api/routes/library.ts` (заменяет `skills.ts`):
  - `GET /api/library?type=&scope=&projectId=` — список.
  - `GET /api/library/:id` — итем + контент.
  - `POST /api/library` — создать (type, scope, slug, title, description, content, frontmatter, agent_filter).
  - `POST /api/library` / `PUT /api/library/:id` — после сохранения вызвать `syncItem(item)`.
  - `DELETE /api/library/:id` — перед удалением записи вызвать `unsyncItem(item)`.
  - `POST /api/library/:id/overrides/:agentType` — задать override-тело → ресинк итема.
  - `POST /api/library/:id/assets` — загрузка вложения (multipart, см. существующий `upload.ts`).
  - `POST /api/library/sync` — ручной полный ресинк (`syncAll`) во все папки агентов.
- Подключить роутер в `server/api/server.ts`; убрать старый `skillsRouter` (или оставить тонкий alias на время миграции).

### Этап 6 — Frontend: раздел «Библиотека»
- Новый view `client/src/views/LibraryView.vue` + маршрут в `client/src/router/index.ts`.
- Pinia store `client/src/stores/library.ts`: загрузка/CRUD итемов.
- UI:
  - Вкладки/фильтр по типу: **Скиллы / Команды / Главные файлы**.
  - Переключатель уровня: **Глобальные / Проектные** (для проектных — выбор проекта).
  - Список итемов с `title` + `description` + статусом enabled.
  - Редактор: поля `name`/`slug`, `description`, markdown-тело, чекбоксы frontmatter (`disable-model-invocation`), выбор «к каким агентам применять» (`agent_filter`), вкладка override-контента по агентам.
  - Кнопка «Синхронизировать всё» → `POST /api/library/sync`.
- В карточке агента — редактор `layout_profile` (можно простой JSON-textarea на первой итерации).

### Этап 7 — Миграция и чистка
- Одноразовая миграция: `data/skills/common.md` → global skill-итем; каждый `data/skills/projects/<id>.md` → project skill-итем для соответствующего проекта.
- Удалить мёртвый код (`deploySkillsToTemp`), старый `skills/manager.ts` после переноса.
- Обновить память проекта (memory) при необходимости.

## Критерии готовности (проверка)

1. Создаю глобальный скилл в UI и сохраняю → файл сразу появляется в `~/.claude/skills/<name>/SKILL.md` и `~/.agents/skills/<name>/SKILL.md`.
2. Создаю проектный скилл → сразу появляется в `{repo}/.claude/skills/...` (и `.agents` для Codex) и лежит постоянно; агент при любом старте его видит.
3. Удаляю скилл в UI → его файлы исчезают из всех папок агентов; чужие файлы пользователя в `.claude/`/`.agents/` остаются нетронутыми.
4. Команда появляется как `/slash` в Claude; у Codex — как скилл.
5. Главный файл раскладывается как `CLAUDE.md` для Claude и `AGENTS.md` для Codex.
6. Динамическая загрузка работает: агент видит `description` и подгружает тело по релевантности (проверить вручную запуском реального агента).
7. Новый CLI-агент с кастомным `layout_profile` получает скиллы по заданным путям.
8. Включаю «Скрыть от git» у проекта → пути попадают в `{repo}/.git/info/exclude` (между маркерами `plangent`), `git status` их не показывает, в `.gitignore` ничего не пишется; выключаю → блок исчезает.

## Важные предостережения для исполнителя

- **Никогда не удалять/перезаписывать чужие файлы пользователя.** Трогать только свои (префикс `plangent-` / своя под-папка).
- Запись глобальных в `~/.claude`/`~/.agents` влияет на всю машину пользователя; делать идемпотентно и только для своих файлов.
- Sync должен быть идемпотентным: повторный вызов не плодит дубли и не ломает существующее.
- Запуск агента (`runs.ts`, `terminal.ts`) в части скиллов больше НЕ участвует — не возвращать старую логику deploy/cleanup.
- Сохранять обратную совместимость API на время миграции, не ломать текущий запуск задач.
