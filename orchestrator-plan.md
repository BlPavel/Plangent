# Оркестратор разработки с ИИ-агентами — план

## Что мы строим

Не «Cursor для команды» и не облачную платформу, а **оркестратор разработки
для одного разработчика**. Его задача — вести устойчивый процесс:

> план → выполнение → обновлённый план → можно продолжить хоть через неделю,
> хоть другой моделью.

**Главная ценность — не в интеграциях, а в устойчивости процесса.**
Память живёт не в сессии агента, а в плане. Поэтому исполнитель сменный:
сегодня Claude Code, завтра Codex, послезавтра Gemini CLI — система не ломается.

Агент — это **одноразовый исполнитель**:

```
этап → новый запуск → прочитал план → выполнил → отметил результат → завершился
```

Это надёжнее длинных сессий (контекст не разрастается, агент не путается,
стоимость не растёт, качество не падает).

---

## Ключевые принципы (на всех этапах)

- **Память — в плане, а не в сессии.** План (markdown с чекбоксами) — единственный
  носитель состояния между запусками.
- **Исполнитель сменный.** Всё работает через адаптер инструмента; Claude/Codex/Gemini
  взаимозаменяемы.
- **Прогресс не доверяем только чекбоксам.** Дополнительно ведём историю запусков
  (кто, что, когда делал) — защита от ошибочно отмеченных пунктов.
- **Терминал — это окно в реальный процесс (PTY), а не «умная сессия».**
  Никакого восстановления чатов, истории за 3 дня, мультидевайса.
- **Только API наружу.** Никакой браузерной автоматизации.
- **Не обобщаем раньше времени.** Делаем под свой стек, подключаемость закладываем
  в интерфейсы, но реализуем по одному модулю.

---

## Модель данных (5 сущностей)

```
Project   пути к репозиториям, Jira, Confluence, настройки
Task      номер задачи
Plan      markdown-файл с этапами (чекбоксы)
Run       запуск агента (кто, когда, что выполнил)
Agent     Claude / Codex / Gemini
```

Пример состояния задачи:

```json
{
  "task": "MSG-123",
  "runs": [
    {
      "agent": "claude",
      "date": "...",
      "completed": ["backend", "dto"]
    }
  ]
}
```

Как это выглядит для пользователя:

```
Проект: Messenger Frontend
Задача: MSG-123

План:
[x] Проанализировать текущую авторизацию
[x] Добавить новый API
[x] Обновить DTO
[ ] Обновить UI
[ ] Написать тесты
[ ] Создать MR

Текущий исполнитель: Claude Code
```

И всё. Никаких живых логов за 3 дня, восстановления чатов, мультипользователя.

## Про живой терминал (снят как риск)

Раньше считали его самым рискованным блоком — по недоразумению. Риск высок только
если строить «умную сессию агента» (восстановление после перезапуска, подключение
с другого компа, история отдельно от терминала, один экран из веба и десктопа).

**Нам это не нужно.** Нам нужен **терминал как окно в реальный процесс**:

```
cd project
export HTTPS_PROXY=...
claude --dangerously-skip-permissions
```

Дальше — показать stdout/stderr в UI, отправлять ввод в stdin. Это обычная работа
с PTY (так устроены терминалы в VS Code, Cursor, JetBrains, ttyd, wetty).
Для прототипа достаточно `xterm.js` + PTY через websocket.

**Упрощение для жизнеспособности:** запускать агента внутри `tmux` / `screen`.

```
tmux new-session -d -s task-123
tmux send-keys ...
tmux capture-pane ...
```

Это даёт почти даром: сессия живёт независимо от UI, можно переподключиться,
открыть тот же терминал вручную, пережить падение приложения, держать десятки сессий.

**Вывод:** живой терминал = низкий риск.

## Где настоящая сложность

80% сложности — не в терминале, а в цепочке контекста:

> выбрать проект → выбрать задачу из Jira → подтянуть нужную документацию →
> собрать контекст → выбрать исполнителя → загрузить навыки →
> вести артефакты между этапами → создать MR

Открытые вопросы здесь (решаем по мере приближения, не заранее):
- как искать нужные страницы Confluence;
- как связывать Jira и документацию;
- как хранить планы;
- как переключать этапы;
- как описывать пайплайны проекта.

Поэтому интеграции добавляем **после** того, как заработает ядро (план + запуск + история).

## Дорожная карта

### Версия 0.1 — Ядро: оболочка вокруг агентов

Доказать, что устойчивый процесс работает. Без Jira, Confluence, GitLab.

- [ ] Модель данных: `Project`, `Task`, `Plan`, `Run`, `Agent` (хранение в файлах + SQLite)
- [ ] Выбор проекта (конфиг: пути к репозиториям, инструмент по умолчанию)
- [ ] Адаптер инструмента: единый интерфейс (запустить / ввод / вывод / завершить /
      разложить скиллы), реализации для Claude Code и Codex
- [ ] Запуск Claude/Codex через PTY (с прокси и нужными флагами), внутри tmux/screen
- [ ] Живой терминал в UI (xterm.js + websocket → PTY), локально
- [ ] Единые skills: общие + проектные, раскладка под инструмент во временную область
      вне репозитория (в проекте папка не появляется)
- [ ] Сохранение плана (markdown с чекбоксами) в хранилище приложения
- [ ] История запусков (`Run`): кто, что, когда выполнил
- [ ] **Сменный исполнитель:** прерванный план продолжается другим агентом — он читает
      план + историю и понимает, что уже сделано

**Готово, когда:** выбрал проект → запустил любым из двух инструментов с едиными
скиллами → агент составил/выполнил план → отметил прогресс → завершился → продолжил
другим инструментом с того же места.

### Версия 0.2 — Jira

- [ ] Модуль Jira (API, под VPN; авторизацию проверить отдельным скриптом заранее)
- [ ] Загрузка задачи по ключу (заголовок, описание, поля) в контекст
- [ ] План привязан к задаче (несколько связанных задач — один план)
- [ ] Автозаполнение имени ветки из шаблона в конфиге (пока как поле)

**Готово, когда:** выбрал задачу из Jira → агент получил контекст → составил план,
привязанный к задаче.

### Версия 0.3 — Confluence

- [ ] Модуль Confluence (API, под VPN): выгрузка страниц в хранилище, кнопка «обновить доку»
- [ ] Поиск по документации (начать с простого, наращивать по надобности)
- [ ] Подгрузка контекста по запросу: агент получает релевантный кусок, а не всю доку

**Готово, когда:** обновил доку под VPN → при работе агент сам подтянул нужные
страницы → контекст не распух.

### Версия 0.4 — GitLab (замыкаем цикл)

- [ ] Модуль GitLab (API, под VPN): авторизация
- [ ] Создание ветки по имени из 0.2
- [ ] Создание MR по команде «готово»
- [ ] (Опционально здесь же) движение тикета в Jira + уведомление в мессенджер

**Готово, когда:** полный сквозной цикл на реальной задаче:
задача → план → реализация → ветка → MR. Первая настоящая польза.

### Версия 1.0 — Командное использование (только если докажет ценность)

- [ ] Сервер состояния (проекты, конфиги, скиллы, планы, статусы)
- [ ] Разделение сервер ↔ локальный раннер (исполнение локально: код, VPN, прокси)
- [ ] Веб-интерфейс (окно в состояние; нужность пересмотреть — возможно, не требуется)
- [ ] Живой терминал через веб (самое хрупкое — осознанно в самый конец)
- [ ] Несколько машин, командное использование

> Примечание: веб-версия и мультидевайс — под вопросом. Для одного разработчика,
> возможно, не нужны вовсе. Решать после 0.4.

## Главное правило

Доведи **один цикл до конца на одной реальной задаче**, прежде чем что-либо обобщать.
Один прогон «задача → план → код → MR» на боевом таске даст больше понимания,
чем месяц проектирования.

Самая жизнеспособная часть всей концепции — устойчивый процесс
«план → выполнение → обновлённый план → продолжить хоть через неделю другой моделью».
Начинаем с неё (версия 0.1). Всё остальное — наращивание вокруг работающего ядра.
